# Multi File Codex Handover Skill

## 何时使用本 Skill

当用户希望用 Codex 开发项目，并且要求：

- 使用三文件交接系统。
- Markdown 交接文档保持中文。
- 新项目需要从 0 建档。
- 已有项目需要读取并建立交接基线。
- 每个线程结束时需要更新交接状态。
- 新线程开始时需要读取旧上下文继续工作。
- 项目允许根据需求安装第三方库，但依赖变更必须记录、验证、可追踪。

## 三文件分工

只维护以下三个交接文件：

```text
AGENTS.md
docs/codex-project-map.md
docs/codex-handoff.md
```

分工如下：

- `AGENTS.md`：项目固定规则、工作方式、禁止事项、第三方依赖规则、测试与验证要求。
- `docs/codex-project-map.md`：项目地图、技术栈、目录结构、入口文件、核心模块、命令、配置文件、依赖清单。
- `docs/codex-handoff.md`：当前任务、最近修改、依赖变更、验证情况、已知问题、下一步建议。

## 总原则

- 所有说明性内容使用中文。
- 代码路径、命令、包名、库名、API 名称、配置项、错误信息保留英文原文。
- 不要把 `AGENTS.md` 写成长篇项目分析。
- 不要把 `docs/codex-handoff.md` 写成历史流水账。
- 每个线程结束时，优先覆盖更新 `docs/codex-handoff.md`。
- 如果项目结构、核心依赖、常用命令或重要配置变化，再更新 `docs/codex-project-map.md`。
- `AGENTS.md` 只在固定规则变化时更新。
- 不能确认的信息要明确写“尚未确认”。
- 不要编造项目结构、命令或依赖。

## AGENTS.md 标准结构

```md
# AGENTS.md

## 工作规则
## 优先阅读
## 禁止事项
## 第三方依赖规则
## 代码规范
## 测试与验证
## 最终回复格式
```

## 第三方依赖规则

在 `AGENTS.md` 中必须写入：

```md
## 第三方依赖规则

- 可以根据任务需要安装第三方库。
- 优先复用项目已有依赖。
- 不要为了小问题引入大依赖。
- 不要同时引入多个功能重叠的库。
- 优先选择成熟、维护活跃、许可证清晰、生态常见的库。
- 对普通低风险依赖，可以在判断必要后直接安装并继续实现。
- 对以下高风险依赖，安装前必须先给出方案并等待用户确认：
  - 安全
  - 加密
  - 鉴权
  - 支付
  - 文件上传
  - 网络请求核心层
  - 数据库迁移
  - ORM
  - 状态管理核心替换
  - 构建系统替换
  - 大型 UI 框架替换
- 每次新增、删除、升级或替换依赖，必须记录：库名、版本、安装命令、引入原因、使用位置、替代方案、验证方式、是否修改 lock 文件。
- 每次依赖变更必须写入 `docs/codex-handoff.md` 的“依赖变更”。
- 如果是核心依赖，还必须更新 `docs/codex-project-map.md` 的“依赖清单”。
```

## 场景一：从 0 开始的新项目

用户说“新项目”“从 0 开始”“建立交接”“创建三文件交接”时：

1. 进入项目建档模式，不进入开发模式。
2. 创建或更新 `AGENTS.md`、`docs/codex-project-map.md`、`docs/codex-handoff.md`。
3. 不修改业务代码。
4. 不修 bug。
5. 不实现功能。
6. 不全仓库搜索。
7. 不安装新依赖。
8. 不格式化全仓库。
9. 不删除文件。
10. 不提交 commit。
11. 只读取理解项目骨架所需的最小文件集。

优先读取：`README`、项目配置文件、依赖声明文件、构建/测试/lint/typecheck 配置、CI 配置、根目录结构最多 3 层、明显入口文件、路由文件、主配置文件。

不要读取：`node_modules/`、`dist/`、`build/`、`coverage/`、`.next/`、`.turbo/`、`target/`、`out/`、`vendor/`、大型 lock 文件完整内容、自动生成文件、二进制文件、大型日志文件。

初始化 `docs/codex-handoff.md` 时，使用：

```md
# Codex 任务交接

## 当前任务
无。

## 上次完成的工作
已初始化 Codex 三文件项目工作基线。

## 最近修改的文件
- AGENTS.md
- docs/codex-project-map.md
- docs/codex-handoff.md

## 依赖变更
无。

## 当前实现状态
未修改业务代码。

## 验证情况
未运行。原因：本线程未修改业务代码。

## 已知问题与风险
暂未确认。

## 下一步建议
开启新线程处理具体任务。

## 下个线程建议优先阅读
- AGENTS.md
- docs/codex-project-map.md
- docs/codex-handoff.md
```

完成后回复：创建或修改的文件、读取的关键文件、判断出的技术栈、当前关键第三方库、后续新线程启动方式、尚未确认的信息。

## 场景二：已有项目分析

用户说“已有项目”“接手旧项目”“读取项目”“分析现有项目”时：

1. 进入项目读取与三文件建档模式。
2. 创建或更新三文件交接系统。
3. 不修改业务代码。
4. 不修 bug。
5. 不重构。
6. 不安装新依赖。
7. 不删除文件。
8. 不提交 commit。

优先读取：`AGENTS.md`、`docs/codex-project-map.md`、`docs/codex-handoff.md` 如果存在，`README`、项目依赖配置、构建/测试/lint/typecheck 配置、CI 配置、根目录结构最多 3 层、明显入口文件、路由文件、主配置文件。

重点识别：项目技术栈、包管理器、构建和运行方式、测试/lint/typecheck/build 命令、核心模块、第三方库体系、功能重叠库、过时或风险依赖、后续新增依赖应优先复用哪些已有库。

完成后回复：读取的文件、创建或更新的文件、当前技术栈、当前核心第三方库、可能的依赖风险、后续新线程启动方式、尚未确认的信息。

## 场景三：线程结束总结

用户说“线程结束”“收尾”“总结”“更新交接”时：

1. 不继续开发新功能。
2. 总结当前线程成果。
3. 必须更新 `docs/codex-handoff.md`。
4. 如果新增了核心依赖、重要模块、新命令或重要配置，也同步更新 `docs/codex-project-map.md`。
5. 除非固定规则变化，否则不要更新 `AGENTS.md`。
6. 不把 `docs/codex-handoff.md` 写成历史日记。

必须记录：本线程完成的工作、修改过的文件、依赖变更、每个依赖变更的库名/版本/安装命令/引入原因/使用位置/替代方案/是否修改 lock 文件/验证结果、已运行验证、未运行验证及原因、仍存在的问题风险或 TODO、下个线程建议优先读取的文件。

最终回复格式：

```md
## 修改文件
- ...

## 依赖变更
- ...

## 验证情况
- ...

## 当前状态
- ...

## 下一步建议
- ...
```

## 场景四：新线程背景续接

用户说“新线程”“继续任务”“基于交接继续”时：

1. 先读取 `AGENTS.md`、`docs/codex-project-map.md`、`docs/codex-handoff.md`。
2. 不重新全仓库分析。
3. 不读取无关目录。
4. 只读取和本任务相关的最小文件集。
5. 复杂任务先给 plan，不要直接修改代码。
6. 不修改无关文件。
7. 不大范围重构。
8. 可以根据需求安装第三方库。
9. 安装第三方库前，先检查项目是否已有相同或相近功能的依赖。
10. 不为了小问题引入大依赖。
11. 引入新库后必须运行最小必要验证。
12. 结束时必须更新 `docs/codex-handoff.md`。
13. 如果引入核心依赖，也要更新 `docs/codex-project-map.md` 的“依赖清单”。
14. 所有交接内容必须使用中文。

普通低风险依赖可以判断必要后直接安装并继续实现。

高风险依赖安装前必须先给出方案并等待确认，包括安全、加密、鉴权、支付、文件上传、网络请求核心层、数据库迁移、ORM、状态管理核心替换、构建系统替换、大型 UI 框架替换。

开始执行前，先输出：

```md
## 任务理解

## 准备读取的文件

## 初步计划

## 是否需要第三方库

## 验证方案
```

## 默认回复风格

- 中文。
- 短句。
- 工程化。
- 先说明行动边界，再执行。
- 不能确认的信息要明确写“尚未确认”。
- 不要编造项目结构、命令或依赖。
